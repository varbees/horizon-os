Backend Stack Decision

Firebase vs server vs serverless — the right answer for a 90-second AI research pipeline with offline-first mobile, push notifications, and Cloudflare R2 storage.

Final Decision
Hybrid, not a single platform. Firebase handles auth + FCM push + realtime community sync. Cloud Run handles the AI pipeline (already built, stateless, no timeout limits). Cloud Tasks queues the 45–90s research job so the app gets an immediate response, then a push notification when done. Cloudflare R2 stores all images and reports (already decided, zero egress). Supabase provides PostGIS geo queries for the species map. Isar in Flutter is the offline-first local database — everything works without signal.

Why not just Firebase

NeedFirebase can do it?Reality
Auth (Google sign-in) Yes Firebase Auth is perfect here — keep it
Push notifications (research ready) Yes FCM is best-in-class, free, already works with Flutter
Realtime community species updates Yes Firestore real-time listeners — keep it for community layer
Run 90s Gemini research agent No Cloud Functions = 60s max. Research pipeline times out. Dead end.
PostGIS geo queries (species within 2km) No Firestore has no native geo radius query. Workaround is painful.
Store images in Cloudflare R2 No Firebase Storage is a different product — can't point to R2
Async job queue (photo → identify → research) Partial Pub/Sub can trigger it but still hits timeout. Cloud Tasks is correct.
Offline-first with Flutter sync Partial Firestore offline works but Isar is faster and works fully without auth
02 — The six pieces

What each service does, one job only

Firebase
Auth + FCM + Firestore
Identity · Push · Community sync

Three things only. Never use Firebase Storage or Functions. Google Sign-In for auth. FCM for "your plant report is ready" notifications. Firestore for community observations that need realtime sync across users.

Firebase Auth → Google Sign-In, zero friction
FCM → research-complete push notification
Firestore → community species sightings feed
Firebase Analytics → funnel tracking
Free tier: generous — 10k auth/month, 50k FCM/day, 1GB Firestore
Cloud Run
FastAPI AI Pipeline
Plant ID · Research agent · Report generation

Already built. Stateless containers. No timeout — Cloud Run lets jobs run as long as needed (up to 3600s). Scales to zero between sessions. The research agent lives here exclusively.

POST /identify → Gemini Vision plant ID
POST /research → Claude/Gemini research agent
POST /notify → fires FCM via Firebase Admin
GET /report/{id} → fetch completed report JSON
Free tier: 2M requests/month, 360k vCPU-seconds — plenty for pilot
Cloudflare R2
Object Storage + CDN
Images · Reports · Classical texts

Already decided. Zero egress fees means serving 100,000 plant images costs the same as serving 1. Workers for image resize on the edge — thumbnail variants auto-generated. Global CDN with cache-control headers.

plants/ → original field photos + compressed versions
reports/ → generated PDF + JSON per species
texts/ → classical text PDFs (Charaka, Agastiyar)
R2 Worker → auto-resize on first request, cache forever
Free tier: 10GB storage, 1M Class-A ops, zero egress — indefinitely
Supabase
PostgreSQL + PostGIS
Species geo database · Trail data

Serverless Postgres with PostGIS extension. The geo layer that Firestore can't do — "show me all documented species within 2km of my GPS." Also stores trail history, observation metadata, guardian rankings.

species table → scientific name, rarity, geo point
observations → user + species + GPS + timestamp
trails → GPS polyline per session
guardians → most-observations-per-species ranking
Free tier: 500MB DB, 2GB bandwidth, 50MB file storage — sufficient for pilot
Cloud Tasks
Async Job Queue
The 45–90s research window

The missing piece that makes Zomato-level notifications possible. App submits photo → gets job_id immediately (< 1s) → Cloud Tasks runs the research in background → when done, fires FCM. User can close app and get notified when ready.

Task created: {job_id, image_url, gps, user_fcm_token}
Cloud Tasks calls Cloud Run /research endpoint
On completion → POST to /notify with job_id + result
Retry logic built-in (3 attempts, exponential backoff)
Free tier: 1M tasks/month — more than enough
Isar (Flutter)
Offline-First Local DB
Everything works without signal

The mountain has no signal in half its zones. Isar stores the full species atlas locally. Photos queue locally. Trail data writes locally first, syncs when signal returns. App is 100% functional offline — sync is a background bonus.

LocalSpecies → full species card cached after first view
PendingCapture → photo queue, uploads when online
TrailSession → GPS points written at 5s intervals
SyncQueue → any write that needs to reach Supabase
Cost: zero. Runs entirely on device. No egress ever.

04 — All Gemini capabilities

Use every model, every feature

The Google AI Studio free tier gives you access to the full Gemini stack. Google Search grounding is the biggest differentiator — it's what makes PlantSage research better than anything Claude alone can do for current, web-verified plant data.

gemini-2.5-flash
Plant visual identification

Fast, cheap, handles images well. First-pass ID. If confidence > 0.7 → accept. If 0.4–0.7 → escalate to Pro. Sub-5s response time. Accepts JPEG directly from camera.

Free: 1500 req/day · 15 RPM · Vision included
gemini-2.5-pro
Deep research agent + hard IDs

The research brain. Better reasoning across 8 domains. Use for the research pipeline and for low-confidence re-identification. Slower but significantly more accurate on ambiguous species. Long context handles full classical texts.

Free: 50 req/day · 2 RPM (sufficient for pilot)
gemini-2.5-pro + Google Search grounding
Real-time web research (the secret weapon)

This is the Perplexity-level research. Google Search grounding lets Gemini search the web during the research call — PubMed, eFlora India, ethnobotany papers, Ayurvedic databases — all current, all cited. No need to manage search APIs separately.

Grounding: included in Gemini API · 1500 searches/day free
text-embedding-004
Classical text semantic search

Embed Charaka Samhita, Agastiyar texts, Bhavaprakasha into Supabase pgvector. When a species is identified, auto-search all classical texts for that plant's Sanskrit/Telugu/Tamil names. Returns exact verse + chapter reference. The ancient knowledge layer.

Free: 1500 req/day · 2048 dimensions
gemini-2.5-flash + audio
Voice field notes transcription

On mountain, hands dirty, can't type. Speak your field notes into the app — Gemini transcribes and structures them as observation data. Elder interviews: record, transcribe, extract plant names + preparation methods automatically into the database.

Free: audio input included in 1500 req/day
Gemini Live API
Real-time plant conversation (future)

Point camera at plant → speak "what is this and can I eat it?" → real-time streaming answer. Not for v1 — but the architecture supports it. The "Star Trek computer for the forest" mode. Stage 3 of the UX plan.

Preview: free in AI Studio · billing when GA
The Google Search grounding advantage: When Gemini searches for "Wrightia tinctoria Andhra Pradesh ethnobotany Yanadi tribe" with grounding enabled, it gets actual current papers, eFlora India entries, and PubMed results — not training data from 18 months ago. This is why the research quality is closer to Perplexity than to a static LLM. And it's free on the AI Studio tier.

OFFLINE SYNC FLOW lib/core/sync_service.dart
// PlantSage Sync Service — offline-first, sync-when-able
// Every write goes to Isar first. Background sync handles the rest.

class SyncService {
  
  // Called every time connectivity changes
  Future<void> onConnectivityRestored() async {
    
    // 1. Upload queued photos to R2
    final pending = await _isar.pendingCaptures
      .filter().uploadedEqualTo(false).findAll();
    
    for (final capture in pending) {
      final url = await _r2.upload(capture.imageBytes, capture.metadata);
      
      // 2. Submit async identify job → get job_id back in <1s
      final job = await _api.submitIdentifyJob(
        imageUrl: url,
        gps: capture.gps,
        fcmToken: await _fcm.getToken(),
      );
      
      // 3. Store job_id locally — poll or wait for push
      capture.jobId = job.id;
      capture.status = CaptureStatus.processing;
      await _isar.writeTxn(() => _isar.pendingCaptures.put(capture));
    }
  }

  // Called when FCM push arrives with completed report
  Future<void> onReportReady(String jobId) async {
    final card = await _api.fetchReport(jobId);
    
    // Cache full species card locally — works offline forever after
    await _isar.writeTxn(() => _isar.localSpecies.put(
      LocalSpecies.fromCard(card)
    ));
    
    // Navigate to card reveal if app is in foreground
    if (_appState.isForegrounded) {
      _router.push('/card/reveal', extra: card);
    }
  }
}
ISAR SCHEMA lib/models/local_species.dart (key collections)
// Everything the app needs, offline, forever once downloaded

@collection
class LocalSpecies {
  Id id = Isar.autoIncrement;
  late String scientificName;   // "Wrightia tinctoria"
  late String teluguName;        // "అడవిపాల"
  late String rarityTier;        // "common" | "rare" | "ancient" | "legendary"
  late String imageUrl;          // R2 CDN URL
  late String reportJsonUrl;     // Full SpeciesCard JSON from R2
  late double lat, lng;          // First documented location
  late DateTime cachedAt;        // For cache invalidation
  bool isGuardian = false;       // User has most observations of this species
}

@collection
class PendingCapture {
  Id id = Isar.autoIncrement;
  late List<int> imageBytes;      // Raw photo bytes until upload
  late double lat, lng;
  late String trailSection;       // "wrightia_corridor" | "granite_ridge" etc
  late DateTime capturedAt;
  bool uploaded = false;
  String? jobId;                  // Set after Cloud Tasks submission
  CaptureStatus status = CaptureStatus.pending;
}

@collection
class TrailSession {
  Id id = Isar.autoIncrement;
  late DateTime startedAt;
  late List<GpsPoint> points;    // Written every 5s
  late double distanceMeters;
  bool synced = false;
}
